# 6×6 リバーシ 仕様書

本書は本アプリケーション（6×6 リバーシ攻略・検討用 Web サイト）の仕様を定める。

## 1. 概要

6×6 盤のリバーシに「マーク石とのコンボによるダメージ」「HP 制の勝敗」を加えた
対戦・検討用の静的 Web サイト。ブラウザのみで動作し（ビルド不要）、
αβ探索による最善手解析を備える。

## 2. ゲームルール

### 2.1 盤面と着手

- 盤は 6×6。座標は列 a–f（左→右）、行 1–6（上→下）で表す（例: d5）。
- 初期配置は中央 4 マス。c3(白)・d3(黒)・c4(黒)・d4(白)。**黒が先手**。
- 着手・反転のルールは標準リバーシに従う（相手の石を一直線に挟むと反転）。
- 打てる場所がない手番は**自動でパス**する（与ダメージ 0）。

### 2.2 マーク

- 初期盤面（中央 4 石）以降に打たれた石には**マーク**が付く。
  表示は黒石 = 🐱、白石 = 🐶。
- **石が反転されるとマークは消える**（以後復活しない）。
  したがってマーク石は常に「打った人の色のまま生き残っている石」である。

### 2.3 コンボと 1 手の獲得ポイント

1 手の獲得ポイント（= 相手への与ダメージ）は、**ポイント式 f(R, C, D)** で決まる。

| 変数 | 定義 |
|---|---|
| R | その手で返した（反転させた）相手の石の枚数（R ≥ 1） |
| C | コンボ数 = **打った石と自分のマーク石で相手の石を挟んで返したライン数**。反転ラインの反対側の端（アンカー）がマーク石であるラインを数える |
| D | **確定コンボ**なら 1、そうでなければ 0（定義は 2.4） |

- C の判定はアンカーが反転されないため着手前の盤面で行える。
- 挟まれて返される「間の石」がマーク石かどうかはコンボに関係しない
  （アンカーのマークのみで決まる）。

### 2.4 確定コンボ（D）

打った石 x について、以下の**両方**が成り立つとき D = 1。

1. **返されない**: 相手の次の 1 手では x を反転できない
   （どの合法応手の反転リストにも x が含まれない）。
2. **確実にコンボできる**: 相手がどの合法手を選んでも、その直後に
   「x をアンカーとするコンボ手」（x からある方向に相手の石が 1 枚以上連続し、
   その先が空きマス）が自分に存在する。相手に合法手がない（パス）場合は
   現盤面で同条件を確認する。

- D は**その手で打った石だけ**を判定する。以前から置かれている石は対象外。
- ポイントが入るのは置いたターンのみ。

### 2.5 ポイント式

- 式は JavaScript の数式として自由に記述できる（変数 R, C, D を使用可）。
- **黒と白で別々の式**を設定できる。
- プリセット:
  - **殴りデッキ**（デフォルト）: `1.4 * 1800 * 1.2 ** (R - 1) * 2 ** C`
  - **その他**: `1000 * 1.2 ** (R - 1) + 1800 * (C + 1) + 1800 * D + 700`
- 式を変更すると、**過去の手も含めて**全ポイント・HP が即座に再計算される
  （各着手の (R, C, D) を記録し、表示のたびに現在の式で評価するため）。
- 不正な式（構文エラー・数値にならない式）は適用されず、エラーを表示して
  直前の式を維持する。

### 2.6 HP と勝敗

- 両プレイヤーは**初期 HP**を持つ。デフォルト各 30000、画面から変更可能（1 以上の整数）。
  初期 HP は**黒と白で別々に設定できる**。対局中の変更も即時反映される。
- 獲得ポイントはそのまま**相手の HP を減らす**。HP は 0 で止めず、
  オーバーキル分も負値としてそのまま扱う（表示・評価とも）。
- **終了条件**（いずれかを満たした時点で終了）:
  1. どちらかの HP が 0 以下 → **相手の HP を削り切った側の勝ち（KO）**
  2. 盤面が全て埋まる、または両者連続パス → **残り HP が多い方の勝ち**（同値は引き分け）
- ゲームは **16 手以内に決着する前提**とする（解析の読み地平の根拠）。

## 3. 画面仕様

### 3.1 盤面

- 6×6 のマス。列ラベル a–f、行ラベル 1–6 を表示。
- 石: 黒 / 白の円。マーク石には 🐱（黒）/ 🐶（白）を重ねて表示。
- **打てるマス**: 背景 `rgb(184, 178, 79)`（黄色系）。
- **コンボが成立するマス**（C ≥ 1）: 背景 `rgb(206, 119, 75)`（橙色系）。
- 打てるマスには以下を表示:
  - 右上: その手の獲得ポイント（与ダメージ）。「+2800」形式。
    10,000 以上は「12.5k」のような k 表記。
  - 下部: その手の内訳「R2 C1 D0」。
- **青枠（枠線）のマス**: 解析エンジンの推奨手。
- **青枠の石**: 直前の着手。
- マスをクリックすると着手する。

### 3.2 盤面下のボタン

左から順に:

| ボタン | 動作 |
|---|---|
| ⟲ 最初から | 盤面を初期状態に戻す（初期 HP・ポイント式などの設定は維持） |
| ↩ 1手戻す | 直前の着手を取り消す（自動パスは着手とまとめて戻る）。終局後も使用可 |
| ★ 最善手を打つ | 解析結果の推奨手を着手する（解析完了までは無効） |

### 3.3 サイドパネル

- **スコアカード**: 黒・白それぞれの残り HP（0 未満も負値のまま表示）。
  中央に現在のターン数と消費手数。手番側のラベルを強調表示。
- **設定カード**: 初期 HP 入力（黒・白それぞれ）、新規ゲームボタン。
- **ポイント式カード**: 黒・白それぞれにプリセット選択
  （殴りデッキ / その他 / カスタム）と式の入力欄。
  入力欄を直接編集するとカスタム扱いになる。式エラーはカード下部に表示。
- **最善プレイ解析カード**（3.4）。
- **ルール説明**（折りたたみ）。

### 3.4 最善プレイ解析

- 双方が最善を尽くした場合の帰結を表示する。
  - 読みの範囲内で KO を強制できる場合: 「**黒/白の勝ち確定（KO） ±N**」。
    N は KO 時点の最終 HP 差（黒−白、オーバーキル分を含む）。
    KO が確定していても、勝ち筋の中で**最終 HP 差が最大になる手**を選ぶ。
  - それ以外: 「**最終HP差（黒−白）: +N**」。
    値は**現在の HP 差を含む**（= 現在 HP 差 + 今後の最善ダメージ差）。
    読み切り前の途中経過には「（暫定）」を付す。
- 有利判定（黒有利 / 白有利 / 互角）と推奨手を併記。
- 進捗行に「読み切り/計算中・深さ d/上限・ノード数・所要 ms」を表示。
- **初期局面は推奨手を d5 に固定**し、探索も root を d5 の 1 手に絞って行う
  （初期盤面は対称で初手 4 候補は同価値のため、評価値は全候補を読んだ場合と
  同値のまま探索量が約 1/4 になる）。スコア・進捗は通常どおり表示し、
  推奨手 d5 は探索完了を待たずに表示・着手可能とする。

## 4. 解析エンジン仕様

### 4.1 探索アルゴリズム

- **negamax + αβ枝刈り + 置換表 + 反復深化**（Web Worker 上で実行）。
- 評価値 = 手番側から見た「双方最善時の最終 HP 差」。
  - KO（与ダメ ≥ 相手の残り HP）を強制できる場合は
    ±(WIN + 最終 HP 差)。WIN = 10¹⁵。HP はクリップせず
    オーバーキル分も差に含めるため、勝ち筋の中でも HP 差最大の手が選ばれる。
  - **KO 時のラウンド完結**: 黒は各ラウンド（黒→白の 2 手）の先手のため、
    黒の KO では白の同一ラウンドの 1 手が残っている。この場合、白の最善
    （最大ダメージ）の応手を受けたものとして **2 ターン分の HP 差**を計算する。
    白の KO はラウンドが完結しているためそのまま。
  - 水平線（深さ 0）および盤面決着（両者連続パス）では現在の HP 差を返す。
  - ダメージは HP 状態（myHP / oppHP）として再帰に流れるため、
    αβ の窓は標準形 (−β, −α) を使う。
- パスは 1 手消費として扱う。両者連続パスで終端。
- root の候補手を 1 手に固定して探索できる（初期局面の d5 固定用、3.4 参照）。

### 4.2 読みの深さと打ち切り

- 反復深化で深さ 1, 2, … と読み、深さごとに途中結果を UI へ通知する。
- 深さ上限 = min(空きマス数 × 2 + 1, max(1, **16 − 経過手数**))。
  読みの地平を「ゲーム開始から 16 手目」に固定する。
  地平が固定されているため、読み切れている範囲では双方が最善を
  打ち続ける限り評価値は一貫する。
- 勝敗が確定（|value| > WIN/2）した時点で打ち切り。
- 時間上限は設けない（読み切るまで計算する。反復深化の途中経過が随時表示される）。
- 局面が変わると実行中の Worker を破棄して新しい解析を開始する。
- 性能の目安（初期局面から深さ 16 の読み切り。root を d5 に固定した実測、
  環境により変動）:
  - 殴りデッキ（D なし）: 約 80 万ノード / 約 3 秒
  - その他（D あり）: 約 130 万ノード / 約 7 秒（D 判定が全ノードで走るため重い。
    数手進むと指数的に軽くなり、中盤以降は数秒以内）

### 4.3 置換表

- キー: 盤面（2bit × 36）+ マーク（1bit × 36）+（手番 | パスフラグ）を
  9 文字の文字列にパック。
- 値: { 残り手数, 評価値, 上下界フラグ, 最善手, 手番側 HP, 相手 HP }。
  同一盤面でも HP が異なれば価値が異なるため、**HP まで一致した場合のみ**値を使う。
  最善手はムーブオーダリングのヒントとして常に使う。
- エントリ数が 200 万を超えたらクリア（メモリ保護）。

### 4.4 ポイント式の評価

- 式は文字列として Worker に渡し、`new Function('R','C','D', ...)` でコンパイル。
- R ∈ [1,36] × C ∈ [0,8] × D ∈ {0,1} の全組み合わせを探索前にテーブル化し、
  探索中の式評価コストをゼロにする（黒・白それぞれ）。
- どちらの式も D を使わない（D=0 と D=1 で値が変わらない）場合、
  探索中の確定コンボ判定を丸ごと省略する。
- D の判定は着手後の盤面が必要なため、ムーブオーダリングは D=0 の近似値で行い、
  実際の評価時に判定する。
- **D 判定の高速化**（結果は相手の全応手を列挙する素朴判定と同値）:
  - コンボ可能判定（comboAvailableFrom）が読むのは、x の各レイの
    「相手色の連続＋その先の停止マス（≤8 個）」だけ。相手は自分の色の石を
    返せないため、相手の応手で変化し得るのは**停止マスのみ**。
  - 現状コンボ手あり（base=true）の場合: 停止マスに絡む応手
    （空き停止マスへの着手 / x 側の色の停止マスを返す着手）だけを
    直接列挙して検証する。相手の全手生成は行わない。
  - 現状コンボ手なし（base=false）の場合: 停止マスに絡まない合法応手が
    1 つ見つかった時点で D=0 が確定する（早期脱出）。
  - この最適化で D 使用時の探索オーバーヘッドは約 +90% → 約 +36% に縮小。

## 5. 状態管理の設計

- 信頼できる状態は**着手マスの列**と画面上の設定（初期 HP・ポイント式）のみとし、
  それ以外は**すべて導出値**とする。着手列をリプレイして
  盤面・マーク・手番・自動パス・各着手の (打ち手, マス, R, C, D) を導出し、
  - ポイント = Σ f(R, C, D)（現在の式で評価）、
  - HP = 初期 HP − 相手のポイント計、
  - 終了 = 両者連続パス ∨ 盤面満杯 ∨ いずれかの HP ≤ 0
  を計算する（自動パスは打てない手番から一意に導出できるため記録しない）。
- この設計により、ポイント式・初期 HP を対局途中に変更しても
  過去の手を含めて整合した状態が再計算され、待った（undo）とも矛盾しない。
- undo は着手列の末尾を 1 つ取り除くだけ（前後の自動パスは導出値なので一緒に戻る）。

## 6. ファイル構成

| ファイル | 役割 |
|---|---|
| `index.html` | 画面構造・ルール説明 |
| `style.css` | スタイル |
| `engine.js` | ゲームロジックと探索エンジン（UI と Worker で共有、依存なし） |
| `worker.js` | 探索専用 Web Worker（式のコンパイルと solve 呼び出し） |
| `game.js` | UI 制御（状態管理・描画・イベント・解析依頼） |
| `docs/SPEC.md` | 本仕様書 |
| `.github/workflows/deploy.yml` | GitHub Pages への自動デプロイ |

- Web Worker を使うため `file://` では動作しない。HTTP サーバー経由で配信すること
  （例: `python3 -m http.server 8000`）。

## 7. 公開（GitHub Pages）

- `main` ブランチへの push で GitHub Actions が起動し、リポジトリ全体を
  GitHub Pages に公開する。
- 初回のみ Settings → Pages → Source を「GitHub Actions」に設定する。
